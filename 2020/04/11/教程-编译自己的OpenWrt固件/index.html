为什么要编译自己的OpenWrt固件

虽然你可以在恩山或者是KoolShare上找到自己设备能用的OpenWrt固件，但是：

  1. 这些固件显然不是为你定制的，里面可能有各种各样你永远都用不上的插件，白白浪费你原本就不充裕的性能与空间。

  2. 大多数论坛上的固件的编译者也不会为你负责，后续如果有更新，或者是什么你想要的新功能，甚至是安全漏洞的补丁，他们都没有义务为你编译新版的固件

  3. 有某些及其恶心的固件编译者会向固件中加后门

编译OpenWrt固件的方法

方法 优点 缺点
Github Actions云编译 1.操作便利，对本地环境无要求 2.适合拥有稳定配置文件后定期编译 1.性能、空间受限 2.国内访问比较麻烦 3.编译失败后只能从头再来
本地编译 1.对固件编译全过程的掌控程度高 2.可以获得较快的编译速度（前提是本机性能够强） 1.本地环境配置比较麻烦（使用虚拟机或者Docker可以比较好地解决问题）

获得配置文件

无论你打算采取什么方法进行编译，你都需要一份配置文件，而配置文件则是固件的DNA，它直接决定了固件会被运行在什么平台上，要包含什么驱动，要包含什么插件……

准备本地环境

用VirtualBox之类的虚拟机软件，创建一个Ubuntu Server 18.04 LTS的虚拟机（本地编译的话，那CPU核心与内存尽量给多，硬盘64G给起步）。然后在本机ssh到虚拟机上，进行进一步的配置。

⚠️文章后面的命令都是在虚拟机上运行的⚠️

⚠️从此步开始请全局科学上网⚠️

# 安装编译依赖的软件包
sudo apt update
sudo apt upgrade -y
sudo apt install -y build-essential asciidoc binutils bzip2 gawk gettext git libncurses5-dev libz-dev patch python3.5 unzip zlib1g-dev lib32gcc1 libc6-dev-i386 subversion flex uglifyjs git-core gcc-multilib p7zip p7zip-full msmtp libssl-dev texinfo libglib2.0-dev xmlto qemu-utils upx libelf-dev autoconf automake libtool autopoint device-tree-compiler g++-multilib

拉取代码仓库

想要配置文件，你就需要一个能被配置的东西，也就是OpenWrt的源码仓库

源码仓库名称 亮点 地址 开发分支 稳定分支
OpenWrt 原版 https://github.com/openwrt/openwrt.git master openwrt-19.07
Lean’s OpenWrt source 适配设备多，ShadowsocksR Plus+插件 https://github.com/coolsnowwolf/lede.git master master
Lienol’s Modified OpenWrt source PassWall插件 https://github.com/Lienol/openwrt.git dev-master dev-19.07 
# 拉取仓库
git clone -b 分支名称 地址 openwrt
# 以Lean's OpenWrt source的master分支为例：git clone -b master https://github.com/coolsnowwolf/lede.git openwrt

生成配置文件

# 进入仓库
cd openwrt
# 更新OpenWrt的软件源
./scripts/feeds update -a
# 安装OpenWrt的软件源中的软件包
./scripts/feeds install -a
# 开始配置（请把终端窗口）
make menuconfig
